# 通用小说全链路生成器架构设计

> 设计来源：沿用 MiMo 的核心判断：先理解现有 `novel0` 流水线，再把它升级为“选题情报 + 正文工厂 + 审稿门禁”的通用小说生产系统。  
> 目标：不是重写当前项目，而是在现有 `bible/`、`outlines/`、`progress/`、`prompts/`、`scripts/novel_workflow.py` 基础上，设计一个可复用、可审稿、可拒绝发布、可迁移到其他小说项目的全链路生成器。

---

## 1. 当前系统判断

`novel0` 已经不是简单的“调用模型写正文”，而是一个有工程边界的连载生产系统。

现有主链路：

```text
build_context
  -> scene_card
  -> draft
  -> review
  -> continuity_check
  -> update_record
  -> next_planner
```

现有资产分层：

```text
bible/      静态约束：世界观、人物、能力、文风、禁区
outlines/   规划层：全书大纲、分卷大纲、章节队列
progress/   动态状态：当前上下文、时间线、伏笔、人物状态、字数
prompts/    阶段提示词：每个生产阶段的执行规则
chapters/   产物层：章节卡、草稿、定稿、审稿记录、连续性检查
scripts/    执行层：命令入口、API 调用、断点续传、文件写入
```

主要短板：

1. 审稿有阶段，但还不是硬门禁；低质量内容仍可能流入后续流程。
2. 正文生成偏“一次成章”，中间意图、场景链、商业改写、去 AI 腔没有拆成稳定产物。
3. 选题、榜单、读者情绪、平台规则没有进入结构化情报层。
4. 当前脚本强绑定 `novel0`，还不能自然复用到第二本书、第三本书。
5. 进度文件存在阻塞和状态冲突时，系统能发现问题，但缺少统一的“阻塞处理器”和修复队列。

结论：升级方向应该是“在原系统外侧加通用编排与门禁”，而不是推翻原有七阶段流水线。

---

## 2. 总体目标

建设一个通用小说全链路生成器，支持从一个题材想法到可发布章节候选的完整流程：

```text
市场情报
  -> 题材卡
  -> 项目 Bible
  -> 全书大纲
  -> 分卷大纲
  -> 章节队列
  -> 章节意图
  -> 场景链
  -> 草稿
  -> 商业改写
  -> 去 AI 腔改写
  -> 审稿门禁
  -> 连续性门禁
  -> 发布检查
  -> 人工确认
```

核心原则：

1. AI 可以生成，但不能绕过审稿进入定稿。
2. AI 可以评分，但不能替代人工最终发布判断。
3. 每个阶段必须有明确输入、输出、文件路径和失败条件。
4. 所有可变状态必须写入 `progress/` 或项目等价目录，不能只存在模型上下文里。
5. 脚本默认不覆盖正式正文；任何覆盖必须显式 `--apply` 或 `--overwrite`。

## 3. 三条主流水线

### 3.1 Topic Scout：选题情报流水线

职责：把外部趋势、榜单、读者情绪、人工观察转成结构化题材卡。

输入：

```text
manual_notes/
platform_rankings/
web_articles/
reader_comments/
competitor_books/
```

输出：

```text
intelligence/processed/topic_candidates.jsonl
intelligence/processed/trope_map.json
intelligence/reports/daily_topic_report.md
intelligence/reports/topic_scoreboard.md
```

题材卡字段：

```json
{
  "topic_name": "寿命可视化 + 都市救赎",
  "target_reader": "喜欢都市异能、命运反转、情绪救赎的读者",
  "core_hook": "主角能看见人生余额，救人要消耗自己的寿命",
  "reader_promise": "每个单元都有危机、代价和反转",
  "differentiation": "不是单纯装逼异能，而是命运代价和情感救赎",
  "scores": {
    "reader_demand": 8,
    "platform_fit": 7,
    "serializability": 8,
    "differentiation": 7,
    "compliance_risk": 3,
    "final_score": 7.1
  }
}
```

第一版可以不联网，先允许人工把榜单观察写入 `intelligence/raw/manual_notes/`，脚本负责抽取和打分。

### 3.2 Draft Factory：正文生产流水线

职责：把章节队列拆成可控中间件，减少“一步写章”的漂移。

通用阶段：

```text
chapter_intent
  -> scene_chain
  -> draft_v1
  -> draft_v2_commercial
  -> draft_v3_humanized
  -> final_candidate
```

每一阶段都保存中间产物：

```text
generation/chapter_intent/ch051_intent.json
generation/scene_chain/ch051_scene_chain.json
generation/draft_v1/ch051_draft_v1.md
generation/draft_v2_commercial/ch051_draft_v2.md
generation/draft_v3_humanized/ch051_draft_v3.md
generation/final_candidate/ch051_final_candidate.md
```

关键约束：

1. `chapter_intent` 只允许有一个本章核心任务。
2. `scene_chain` 必须是 3-5 个有因果关系的场景。
3. `draft_v1` 只负责完整讲清剧情，不追求最终文风。
4. `draft_v2_commercial` 强化冲突、爽点、追读钩子。
5. `draft_v3_humanized` 删除 AI 腔、抽象总结句、说明书式设定解释。
6. `final_candidate` 只是候选，不直接进入 `chapters/final/`。

### 3.3 Review Gate：审稿门禁流水线

职责：判定章节是否可以发布、需要小修，还是必须重写。

审稿维度：

```text
hook_score                     章节钩子
conflict_score                 冲突强度
character_consistency_score    人物一致性
continuity_score               连续性
ai_trace_score                 去 AI 痕迹程度
reader_retention_score         追读潜力
platform_risk_score            平台风险，分数越高风险越大
```

默认门禁：

```text
hook_score < 7                  不可发布
conflict_score < 7              不可发布
character_consistency_score < 8 不可发布
continuity_score < 8            不可发布
ai_trace_score < 7              不可发布
reader_retention_score < 7      不可发布
platform_risk_score > 5         不可发布
fatal_issues 非空              不可发布
```

输出：

```text
reviews/chapter_reviews/json/ch051_review.json
reviews/chapter_reviews/markdown/ch051_review.md
reviews/aggregate/revision_queue.md
reviews/aggregate/repeated_ai_patterns.md
```

## 6. Agent 分工设计

这里的 Agent 不一定是常驻服务，第一版可以是脚本中的角色化 prompt。

### 6.1 Scout Agent

职责：

```text
读取人工情报和榜单观察
抽取题材、设定、开局、标题模式
生成 topic_candidates.jsonl
输出每日选题报告
```

失败条件：

```text
没有明确读者承诺
没有差异化
合规风险过高
无法持续连载 100 章以上
```

### 6.2 Architect Agent

职责：

```text
把题材卡转成项目 Bible
生成世界观、人物、能力、禁区、文风
生成全书大纲和分卷大纲
初始化 progress/
```

失败条件：

```text
核心设定不可连续生产
主角目标不清
能力边界不可控
长期主线无法拆成卷结构
```

### 6.3 Planner Agent

职责：

```text
根据大纲和 progress/ 生成章节队列
每章只定义一个核心任务
维护必须承接和必须避免
```

失败条件：

```text
章节任务模糊
连续两章没有信息增量
提前解决长期主线
忽略未回收伏笔
```

### 6.4 Writer Agent

职责：

```text
执行 Draft Factory
输出章节意图、场景链、草稿、商业改写、人味改写、定稿候选
```

失败条件：

```text
跳过章节意图
场景无冲突
新增未登记核心设定
正文直接写入 final/
```

### 6.5 Editor Agent

职责：

```text
执行 Review Gate
输出 JSON 审稿结果和 Markdown 审稿报告
决定 publishable / minor_revision / rewrite_required
```

失败条件：

```text
评分无理由
JSON 不合法
发现 fatal issues 仍判定可发布
没有给出可执行修改任务
```

### 6.6 Continuity Agent

职责：

```text
检查人物状态、时间线、能力边界、伏笔、前文事实
维护 continuity_index.md 和 blocker_queue.md
```

失败条件：

```text
用新章覆盖旧事实
时间线无法解释
能力越权
人物关系突然跳变
```

### 6.7 Publisher Agent

职责：

```text
汇总章节审稿结果
生成 publish_check_report.md
生成 revision_queue.md
只给发布建议，不自动发布
```

失败条件：

```text
有不可发布章节仍给出整体上线建议
没有列出上线前必须完成项
```

## 7. 命令设计

保留现有命令：

```bash
make check
make status
make run-batch START=31 COUNT=10
make run-full N=31 TITLE=标题
```

新增 v2 命令：

```bash
make scout-topics
make bootstrap-project TOPIC=topic_id
make plan-range START=51 COUNT=10
make generate-v2 CH=51
make review-chapter CH=51
make review-range START=1 END=10
make publish-check START=1 END=10
make repair-blockers
```

命令职责：

```text
scout-topics       只写 intelligence/
bootstrap-project  只初始化新项目骨架，不覆盖已有项目
plan-range         只写 outlines/chapter_queue.md 的候选区
generate-v2        只写 generation/ 和 reviews/
review-chapter     只读章节正文，写 reviews/
review-range       批量审稿
publish-check      汇总上线检查
repair-blockers    根据 blocker_queue.md 给修复建议，默认不改文件
```

## 9. 落地阶段


### Phase 4：阻塞处理器

目标：把当前 `progress/current_context.md` 中的阻塞问题变成可追踪任务。

任务：

```text

3. 输出阻塞清单和修复建议。

5. 加 --apply 后才写入修复后的候选文件。
```

验收：

```bash

```

## 10. 当前 novel0 的推荐执行顺序

当前 `progress/current_context.md` 显示仍有高优先级阻塞，例如第 31-49 章状态、人物状态、能力阶段、时间线抽取不一致。因此不要直接从第 50 章继续生成。

推荐顺序：

```text
1. 先实现 Review Gate，对第 1-30 章或第 1-50 章做 publish-check。
2. 把审稿结果中的共性问题写入 reviews/aggregate/weak_points.md。
3. 建立 progress/character_state.md 的可信版本。
4. 用 repair_blockers.py 把 current_context 的阻塞转成任务。
5. 确认第 31-49 章到底是已完成、待修复，还是要重新生成。
6. 若继续写第 51 章，必须先让第 31-50 章状态统一。
7. 新章节统一走 generate-v2，不再直接 run-full 写 final。
```

## 11. 风险与边界

### 11.1 不做全自动发布

系统可以输出 `publishable: true`，但只代表通过机器门禁。正式发布仍需要人工确认。

### 11.2 不让选题情报污染既有项目

Topic Scout 的结果只能影响新项目或下一轮大纲调整，不能直接改写当前小说已经发生的事实。

### 11.3 不用模型记忆代替状态文件

任何人物变化、能力变化、伏笔新增、时间线变化，都必须落盘到 `progress/`。

### 11.4 不直接覆盖旧正文

新系统默认写：

```text
generation/
reviews/
intelligence/
```

只有人工确认后，才允许把 `generation/final_candidate/` 合并到 `chapters/final/`。

---

## 12. 最小可行版本

如果只做一天，优先做这些：

```text
1. reviews/rubrics/
2. scripts/lib/file_utils.py
3. scripts/lib/llm_client.py
4. scripts/review_gate.py
5. scripts/publish_check.py
6. Makefile 审稿命令
```

原因：当前项目已经能生成章节，最先需要补的是“发布前门禁”和“批量质量报告”。只要 Review Gate 建起来，后面的 Generate v2 和 Topic Scout 都可以逐步接入。

## 13. 成功标准

系统升级完成后，应该能稳定回答以下问题：

```text
这本书现在能不能继续写？
下一章为什么要写这个？
这一章读者看完获得什么？
这一章有没有 AI 味？
这一章有没有破坏前文？
前 10 章能不能上线？
哪些章节必须重写？
哪些问题是重复出现的系统性问题？
如果换一本书，这套流程能不能复用？
```

对应命令：

```bash
make status
make generate-v2 CH=51
make review-chapter CH=51
make publish-check START=1 END=10
make repair-blockers
make scout-topics
```

这就是通用小说全链路生成器的第一版边界：先保证流程可控，再追求效率；先让系统会拒绝，再让系统多写。
