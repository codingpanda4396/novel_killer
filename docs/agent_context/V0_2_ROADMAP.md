# V0.2 Roadmap

> 最后更新：2026-05-03  
> NovelOps v0.2 版本目标规划

## 1. 版本目标概述

v0.2 是 NovelOps 的下一个重要里程碑版本，核心目标是增强系统的智能化、商业化能力和可观测性，重点围绕以下五个方向推进：

1. **Orchestrator（主控编排器）**
2. **网文审稿增强**
3. **Billing（计费系统）**
4. **Analytics（数据分析）**
5. **Dashboard（可视化看板）**

## 2. 详细规划

### 2.1 Orchestrator（主控编排器）

**目标**：构建一个统一的主控编排器，协调各个专业模块完成复杂的创作任务。

**设计原则**（已决策）：
- 采用 **主控 Orchestrator + 专业模块** 架构
- 不做自由多 Agent 对话系统
- Orchestrator 负责任务分解、调度和状态管理
- 专业模块保持独立，专注于各自领域逻辑

**核心功能**：
- [ ] 任务编排：将复杂请求（如"生成下一章并审稿"）分解为子任务序列
- [ ] 状态机：管理创作流程的各个阶段（规划 → 生成 → 审稿 → 修订）
- [ ] 错误处理：模块间调用失败时的重试和降级策略
- [ ] 上下文传递：在任务链中传递项目状态、章节信息等上下文
- [ ] 可恢复性：支持中断后从检查点恢复

**实现建议**：
- 在 `novelops/orchestrator.py` 中实现
- 定义清晰的任务接口和状态数据结构
- 复用现有的 `assistant.py`（意图识别）和 `planner.py`（章节规划）
- 必须包含完整的 unittest 测试覆盖

---

### 2.2 网文审稿增强

**目标**：针对中文网文特点，增强审稿系统的专业性和准确性。

**当前基础**：
- 已有 `reviewer.py` 提供基础审稿功能
- 支持结构化评分、问题列表、修订建议
- 支持分数钳制（0-100）和动作标准化

**增强方向**：
- [ ] **网文专项评分维度**：爽点密度、节奏把控、钩子强度、打脸满意度等
- [ ] **平台适配**：针对不同平台（番茄、起点等）的审稿标准配置
- [ ] **批量审稿**：支持批量审稿和发布前检查（`publish-check` 命令已存在，需增强）
- [ ] **审稿模板**：不同题材（仙侠、都市、玄幻等）的审稿模板和 rubric
- [ ] **审稿解释增强**：更详细地解释为何某章节未通过审稿
- [ ] **审稿历史追踪**：追踪章节审稿历史，识别重复问题

**实现建议**：
- 扩展 `novelops/reviewer.py`
- 在 `novelops/config.py` 中增加审稿模板配置
- 增强 `novelops/readiness.py` 中的平台适配检查

---

### 2.3 Billing（计费系统）

**目标**：追踪和计费 LLM API 的使用成本，帮助用户控制创作成本。

**核心功能**：
- [ ] **API 调用记录**：记录每次 LLM API 调用的模型、tokens、成本
- [ ] **成本追踪**：按项目、按章节、按时间段统计成本
- [ ] **预算告警**：当成本接近预设预算时发出告警
- [ ] **成本报告**：生成成本分析报告（日/周/月）
- [ ] **多模型计费**：支持不同 LLM 提供商（DeepSeek、Claude via RightCode）的计费规则

**数据存储**：
- 在现有 SQLite 数据库中新增表：`api_calls`、`cost_records`
- 或独立计费数据库（视数据量而定）

**实现建议**：
- 新建 `novelops/billing.py` 模块
- 在 `novelops/llm.py` 中埋点记录 API 调用
- 提供 CLI 命令：`billing-report`、`billing-budget-set`

---

### 2.4 Analytics（数据分析）

**目标**：提供创作数据分析，帮助用户了解创作进度、质量和效率。

**核心功能**：
- [ ] **创作进度分析**：章节完成度、字数统计、生成速度
- [ ] **审稿通过率**：按项目、按时间段统计审稿通过率
- [ ] **修订效率**：修订次数、修订耗时分析
- [ ] **内容质量趋势**：评分趋势、问题类型分布
- [ ] **LLM 使用效率**：tokens/章节、成本/章节分析

**数据来源**：
- 现有 SQLite 数据库中的 `chapters`、`reviews`、`revision_queue` 等表
- 新增的 billing 数据

**实现建议**：
- 新建 `novelops/analytics.py` 模块
- 提供 CLI 命令：`analytics-report`
- 支持导出 CSV/JSON 格式报告

---

### 2.5 Dashboard（可视化看板）

**目标**：增强 Web 界面的可视化能力，提供更直观的项目状态和数据展示。

**当前基础**：
- 已有 FastAPI Web 服务（`novelops/web.py`）
- 现有页面：项目列表、项目工作台、章节详情、修订队列
- 使用 Jinja2 模板渲染

**增强方向**：
- [ ] **项目概览仪表盘**：项目进度、审稿通过率、成本统计等关键信息一目了然
- [ ] **数据可视化**：使用 Chart.js 或类似轻量级库绘制趋势图、分布图
- [ ] **实时状态**：WebSocket 或轮询方式展示生成、审稿等长时间任务的进度
- [ ] **移动端适配**：优化移动端浏览体验
- [ ] **深色模式**：支持深色/浅色主题切换

**技术选型**：
- 前端：保持轻量级，优先使用现有 Jinja2 + 少量 JavaScript
- 图表：Chart.js（约 70KB gzipped）或 ECharts（如需要更复杂图表）
- 避免引入 React、Vue 等重型框架

**实现建议**：
- 扩展 `novelops/web.py` 中的路由和 API
- 新增模板：`dashboard.html`、`analytics.html`
- 新建 `novelops/templates/` 下的模板文件

## 3. 优先级与时间规划

| 优先级 | 功能 | 预估工作量 | 依赖 |
|--------|------|-----------|------|
| P0 | Orchestrator 核心框架 | 3-5 天 | 无 |
| P0 | 网文审稿增强（基础） | 2-3 天 | 无 |
| P1 | Billing 基础功能 | 3-4 天 | 需修改 llm.py |
| P1 | Analytics 基础功能 | 2-3 天 | 依赖 Billing 数据结构 |
| P2 | Dashboard 增强 | 3-5 天 | 依赖 Analytics |
| P2 | 网文审稿增强（高级） | 2-3 天 | 依赖审稿基础增强 |
