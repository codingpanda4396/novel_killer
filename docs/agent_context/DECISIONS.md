## DEC-001：采用主控 Orchestrator + 专业模块架构

**日期**：2026-05-03  
**状态**：accepted

### 背景

随着项目功能增多，需要一种有效的方式来协调各个专业模块（生成、审稿、规划等）完成复杂的创作任务。有两种主要架构方向：

1. **自由多 Agent 对话系统**：多个 Agent 自由对话、协商完成任务
2. **主控 Orchestrator + 专业模块**：一个主控器负责编排，专业模块各司其职

### 决策

采用 **主控 Orchestrator + 专业模块** 架构，不做自由多 Agent 对话系统。

### 理由

1. **可预测性**：Orchestrator 模式任务流程明确，易于调试和测试
2. **可控性**：主控器可以精确控制每个步骤，避免自由对话系统可能出现的不可控行为
3. **复用现有模块**：项目已有 `assistant.py`（意图识别）、`planner.py`（规划）、`generator.py`（生成）、`reviewer.py`（审稿）等专业模块，只需新增编排层
4. **测试友好**：专业模块可以独立测试，Orchestrator 可以通过 Mock 对象测试编排逻辑
5. **性能**：避免多 Agent 对话的反复 LLM 调用，降低成本和延迟
6. **复杂度控制**：自由对话系统需要解决 Agent 协商、冲突解决等复杂问题，而 Orchestrator 模式更直观

### 后果

**正面**：
- ✅ 架构清晰，易于理解和维护
- ✅ 各模块职责单一，便于独立开发和测试
- ✅ 任务流程可控，便于错误排查
- ✅ 与现有代码结构自然衔接

**负面**：
- ⚠️ Orchestrator 可能成为单点故障（需做好错误处理）
- ⚠️ 灵活性不如自由对话系统（但可控性更重要）
- ⚠️ 需要仔细设计 Orchestrator 接口和状态管理

### 相关文件

- `novelops/assistant.py` - 意图识别（将被 Orchestrator 调用）
- `novelops/planner.py` - 章节规划
- `novelops/generator.py` - 章节生成
- `novelops/reviewer.py` - 章节审稿
- `V0_2_ROADMAP.md` - v0.2 Orchestrator 实现计划

## DEC-002：保持轻量级依赖，不使用前端框架

**日期**：2026-05-03  
**状态**：accepted

### 背景

Web 界面需要增强可视化能力（Dashboard、Analytics 等）。在选择技术栈时，考虑是否引入现代前端框架（React、Vue 等）。

### 决策

保持轻量级依赖，Web 界面继续使用 Jinja2 服务端渲染 + 少量 JavaScript，不引入前端框架。

### 理由

1. **项目定位**：NovelOps 是工具型应用，不是复杂的 Web 应用
2. **团队规模**：小团队/个人项目，维护成本要考虑
3. **依赖控制**：符合 `CODING_RULES.md` 中"不引入重型依赖"的原则
4. **部署简单**：无需 Node.js 构建步骤，直接运行 Python 服务
5. **性能**：服务端渲染首屏速度快，对内部工具足够

### 后果

**正面**：
- ✅ 部署简单，无需前端构建流程
- ✅ 依赖少，维护成本低
- ✅ 对 Python 开发者友好

**负面**：
- ⚠️ 复杂交互实现成本较高
- ⚠️ 前端代码可能不如框架组织清晰
- ⚠️ 需要手动处理一些前端逻辑

### 妥协方案

对于数据可视化等需要复杂前端交互的场景：
- 使用轻量级图表库（Chart.js，约 70KB gzipped）
- 必要时可考虑 ECharts 精简版
- 保持 JavaScript 代码量在可控范围内

## DEC-003：使用 unittest 而非 pytest

**日期**：2026-05-03  
**状态**：accepted

### 背景

项目需要选择测试框架。主流选择有 unittest（标准库）和 pytest（第三方）。

### 决策

使用 unittest 作为测试框架，不引入 pytest。

### 理由

1. **标准库**：unittest 是 Python 标准库，无需额外依赖
2. **符合轻量级原则**：避免引入不必要的第三方依赖
3. **已有基础**：项目已有 37 个 unittest 测试用例，延续使用
4. **CI 简单**：CI 环境直接运行 `python3 -m unittest discover -s tests` 即可

### 后果

**正面**：
- ✅ 无额外依赖
- ✅ CI 配置简单
- ✅ 与项目现有测试一致

**负面**：
- ⚠️ pytest 的一些特性（如 fixture、参数化）需要通过 unittest 方式实现
- ⚠️ 测试代码可能略显冗长

### 备注

如果未来项目复杂度大幅增加，可以考虑引入 pytest，但目前 unittest 足够。

## DEC-004：SQLite 作为主存储，Markdown/JSON 作为源文件

**日期**：2026-05-03  
**状态**：accepted

### 背景

项目需要存储项目元信息、章节索引、审稿记录等数据。考虑数据库选型。

### 决策

- 使用 **SQLite** 作为查询索引数据库（`runtime/novelops.sqlite3`）
- 正文、bible、outlines 和审稿报告仍以 **Markdown/JSON 原件**为准
- SQLite 只做查询索引，不直接存储正文内容

### 理由

1. **单行部署**：SQLite 无需独立数据库服务，文件即数据库
2. **版本友好**：Markdown/JSON 源文件可以纳入 git 版本控制
3. **查询效率**：SQLite 索引加速项目、章节、审稿等查询
4. **数据分离**：源文件和索引分离，便于备份和迁移
5. **已有实现**：项目已有 `novelops/indexer.py` 管理 SQLite 索引

### 后果

**正面**：
- ✅ 部署简单，无需数据库服务
- ✅ 源文件可版本控制
- ✅ 查询性能好

**负面**：
- ⚠️ 需要维护索引同步（已有 `rebuild_index` 命令）
- ⚠️ 多进程写入需要小心处理（目前是单用户场景，影响不大）

## DEC-005：要求真实 LLM，不提供 no-LLM/mock fallback

**日期**：2026-05-03  
**状态**：accepted

### 背景

早期版本可能考虑在没有 LLM 时提供 mock 模式或降级方案。

### 决策

当前版本已经取消 no-LLM/mock fallback：所有自然语言解析、审稿和生成都要求真实 LLM 可用。

### 理由

1. **功能完整性**：mock 模式无法真实测试 LLM 相关功能
2. **简化代码**：不需要维护 mock 逻辑和 fallback 分支
3. **明确依赖**：用户清楚知道需要配置有效的 LLM API 密钥
4. **测试策略**：测试中使用 Fake LLM 客户端（不是 mock fallback，而是测试策略）

### 后果

**正面**：
- ✅ 代码更简洁
- ✅ 功能行为明确
- ✅ 测试使用 Fake 客户端，不依赖真实 LLM

**负面**：
- ⚠️ 没有配置 LLM 时功能完全不可用
- ⚠️ 新用户需要先获取 API 密钥才能使用

### 备注

测试中使用的 Fake/Mock 对象（如 `FakeClient`、`FakeFrameworkClient`）是测试策略，不是产品的 mock fallback。

## 决策更新流程

当需要做新的架构决策时：

1. 在 `V0_2_ROADMAP.md` 或用户需求中识别需要决策的问题
2. 调研可选方案，列出优缺点
3. 与用户讨论确认（如需要）
4. 在此文件中新增决策记录，状态设为 `proposed`
5. 实施并验证后，将状态更新为 `accepted`
6. 如果后续决策替代了旧决策，将旧决策标记为 `superseded` 并注明替代者
